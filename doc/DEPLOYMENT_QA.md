## 1. 为什么要使用 Nginx？

### 核心结论

**可以不用 Nginx，但生产环境强烈建议使用。**

### 详细解释

| 功能 | NestJS 应用 | Nginx |
|------|------------|-------|
| **静态文件服务** | ⚠️ 效率低 | ✅ 专业处理（效率是 Node.js 的几十倍） |
| **HTTPS 配置** | ⚠️ 配置复杂 | ✅ 简单配置（Certbot 一键获取证书） |
| **负载均衡** | ❌ 不支持 | ✅ 原生支持（可运行多个应用实例） |
| **请求限流** | ⚠️ 需自行实现 | ✅ 简单配置 |
| **端口转发** | ❌ 不支持 | ✅ 原生支持（一个域名代理多个服务） |
| **安全防护** | ⚠️ 需自行实现 | ✅ 专业防护（IP 白名单、防火墙） |

### 典型部署架构

```
用户请求 → Nginx(80/443) → NestJS 应用(3000) → 响应
```

### 什么情况下可以不用 Nginx？

- 开发环境
- 内部测试服务
- 非常简单的单实例部署（不推荐生产使用）

## 2. 蓝绿部署和多环境部署怎么理解？

### 蓝绿部署（Blue-Green Deployment）

**核心思想**：同时运行两套完全相同的环境，通过切换流量实现零停机部署。

```
┌──────────┐      ┌──────────┐
│  蓝环境   │      │  绿环境   │
│ (当前版本) │      │ (新版本) │
└────┬─────┘      └────┬─────┘
     │                 │
     └────────┬────────┘
              ▼
        负载均衡器
              │
              ▼
         用户请求
```

**优势**：
- ✅ 零停机部署
- ✅ 快速回滚（切换回蓝环境即可）
- ✅ 用户无感知

**劣势**：
- ❌ 成本高（双倍资源）
- ❌ 需要同步数据库
- ❌ 配置复杂

### 多环境部署（Multi-Environment Deployment）

**核心思想**：在不同的环境中部署应用，代码需要经过多个阶段的验证才能到达生产环境。

```
开发环境 → 测试环境 → 预发环境 → 生产环境
   │           │           │           │
本地开发     自动化测试   模拟生产     真实用户使用
```

**各环境职责**：

| 环境 | 用途 | 特点 |
|------|------|------|
| **开发环境** | 本地开发、调试 | 数据随意、配置宽松 |
| **测试环境** | 自动化测试、手动测试 | 数据隔离、配置严格 |
| **预发环境** | 模拟生产、最终验证 | 配置与生产一致 |
| **生产环境** | 真实用户使用 | 数据真实、安全优先 |

### 两者的关系

```
多环境部署（代码质量验证）
       │
       ▼
开发 → 测试 → 预发
       │
       ▼
蓝绿部署（生产环境切换）
       │
       ▼
生产环境（安全上线）
```

**最佳实践**：
1. 使用**多环境部署**确保代码质量
2. 在**生产环境**使用**蓝绿部署**实现零停机上线

## 3. 为什么有的 CI 过程会加入 Jenkins？

### Jenkins vs GitHub Actions/GitLab CI

| 维度 | Jenkins | GitHub Actions / GitLab CI |
|------|---------|---------------------------|
| **部署方式** | 自建服务器 | 云端托管 |
| **维护成本** | 高（需自行维护服务器） | 低（平台维护） |
| **灵活性** | 极高（完全自定义） | 中等（平台约束） |
| **集成能力** | 插件丰富，几乎支持所有工具 | 原生集成，生态有限 |
| **学习曲线** | 陡峭 | 平缓 |
| **成本** | 服务器成本 | 免费/按使用付费 |

### 为什么选择 Jenkins？

| 场景 | 说明 |
|------|------|
| **复杂的构建流程** | 需要并行执行多个测试套件、跨平台构建、与企业内部系统集成 |
| **企业级需求** | 私有化部署（数据安全合规）、统一的权限管理、自定义插件开发 |
| **遗留系统迁移** | 已有 Jenkins 基础设施、大量自定义 Pipeline、团队熟悉 Jenkins |
| **高级功能需求** | 分布式构建、蓝绿部署、金丝雀部署等高级部署策略 |

### 你的项目建议

对于当前项目，**GitHub Actions 完全足够**。如果未来需要更复杂的部署策略，可以考虑引入 Jenkins。

## 4. Push main 执行部署，PR 不部署在哪里体现？

### 关键配置

```yaml
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  build-and-test:
    # 无 if 条件，两个事件都执行
    steps:
      - name: Build
        run: pnpm run build

  deploy:
    needs: build-and-test
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'  # ← 关键！
    steps:
      - name: Deploy to server
```

### 核心逻辑

| 场景 | `github.event_name` | `github.ref` | 是否执行 deploy |
|------|---------------------|--------------|----------------|
| push main | `push` | `refs/heads/main` | ✅ 执行 |
| PR to main | `pull_request` | `refs/pull/123/merge` | ❌ 不执行 |

**deploy job 的 `if` 条件决定了是否执行部署**：
- `github.event_name == 'push'`：只在代码推送时执行
- `github.ref == 'refs/heads/main'`：只在 main 分支时执行

## 5. 为什么要分离 build 和 deploy？

### 核心原则：构建一次，到处运行

```
CI 环境构建 → 生成 dist → 上传到任意服务器
```

### 详细原因

| 设计决策 | 原因 | 好处 |
|---------|------|------|
| **分离 build 和 deploy** | 关注点分离 | 易于维护，修改构建流程不影响部署 |
| **CI 环境构建** | 环境一致性 | 每次构建都在标准化环境中进行 |
| **Artifact 传递** | 隔离构建环境 | 构建过程不影响生产服务器 |
| **分步执行** | 易于调试 | 每一步失败都可以单独定位和重试 |
| **不在服务器构建** | 资源隔离和安全 | 减少攻击面，简化服务器维护 |

### 对比：简单方案 vs 当前方案

**简单方案**（在服务器直接构建）：
- ❌ 耦合度高，难以调试
- ❌ 环境不一致，可能"CI 构建成功，服务器构建失败"
- ❌ 构建占用服务器资源

**当前方案**（分离构建和部署）：
- ✅ 环境隔离，易于调试
- ✅ 安全可靠，不暴露源代码到服务器
- ✅ 资源优化，服务器只负责运行

## 6. 为什么 PR 只测试不部署，Push 直接部署？

### PR 和 Push 的本质区别

| 概念 | 本质 | 代码状态 |
|------|------|---------|
| **PR** | 请求合并 | 还在审查阶段，可能有问题 |
| **Push main** | 正式合入主分支 | 已经通过审查，可以安全上线 |

### PR 不部署的原因

| 原因 | 说明 |
|------|------|
| **PR 是草稿** | 代码可能被修改多次，每次都触发 CI 测试，如果每次都部署会导致生产环境频繁变动 |
| **保护生产环境** | PR 的代码可能存在逻辑错误、性能问题、安全漏洞，直接部署会影响用户 |
| **符合开发流程** | 代码需要经过"自动化测试 → 代码审查 → 合并 → 部署"的完整流程 |

### Push 直接部署的原因

| 原因 | 说明 |
|------|------|
| **代码已通过验证** | push main 意味着代码已经经过自动化测试和代码审查 |
| **持续部署效率** | 审查通过后自动部署，无需人工干预，快速交付功能 |

### 核心原则

```
代码审查前 → 只测试不部署（保护生产）
代码审查后 → 自动部署（快速交付）
```

## 7. PR 阶段为什么构建，Push 阶段又要重复构建？

### 当前配置的优化

**优化前**：PR 和 Push 都会上传 artifact，Push 阶段重新下载后部署（没有重复构建）

**优化后**：

```yaml
- name: Upload build artifact
  if: github.event_name == 'push' && github.ref == 'refs/heads/main'  # ← 只在 push main 时上传
  uses: actions/upload-artifact@v4
```

### 执行流程

| 阶段 | 构建 | 上传 artifact | 部署 |
|------|------|--------------|------|
| **PR** | ✅ 构建（验证代码） | ❌ 不上传 | ❌ 不部署 |
| **Push main** | ✅ 构建（一次构建） | ✅ 上传 | ✅ 使用 artifact 部署 |

### 为什么 PR 和 Push 是两个独立的工作流？

- PR 和 Push 是两个独立的 GitHub 事件，触发不同的工作流实例
- PR 阶段的代码可能与 Push 阶段的代码不同（其他 PR 合并、冲突解决等）
- 重新构建可以确保部署的是最新代码，保证代码完整性

## 8. PR 阶段为什么要做一次构建？

### 核心原因：构建本身就是一种验证

```
构建 ≠ 为了部署
构建 = 验证代码是否可以正常编译
```

### 构建能发现的问题

| 问题类型 | 示例 | 测试能发现吗？ | 构建能发现吗？ |
|---------|------|--------------|--------------|
| **语法错误** | `const x == 1;` | ❌ 不能 | ✅ 能 |
| **类型错误** | `const x: string = 1;` | ❌ 不能 | ✅ 能 |
| **缺少依赖** | 忘记安装某个包 | ❌ 不能 | ✅ 能 |
| **路径错误** | `import '../utils'` 路径写错 | ❌ 不能 | ✅ 能 |
| **配置错误** | `tsconfig.json` 配置错误 | ❌ 不能 | ✅ 能 |

### PR 阶段的完整验证体系

```
Lint → Test → Build
代码规范检查 → 功能逻辑测试 → 编译验证
```

三层验证，层层把关，缺少任何一层都可能让问题漏网！

### 提前发现问题的价值

| 场景 | 耗时 | 影响 |
|------|------|------|
| PR 阶段发现问题 | 几分钟 | 开发者立即修复，不影响主分支 |
| Push 阶段发现问题 | 可能几小时 | 需要回滚，影响其他开发者 |

## 9. CI/CD 工作流总结

### 完整流程

```
┌─────────────────────────────────────────────────────────────┐
│                    CI/CD 工作流                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  事件触发                                                    │
│     │                                                       │
│     ├── PR to main                                          │
│     │     │                                                 │
│     │     ▼                                                 │
│     │  build-and-test ✅                                     │
│     │  ├─ Checkout → Node.js → Install → Lint              │
│     │  ├─ Test → Build                                      │
│     │  └─ Upload artifact ❌ 跳过                           │
│     │                                                       │
│     │  deploy ❌ 跳过                                        │
│     │                                                       │
│     └── Push to main                                        │
│           │                                                 │
│           ▼                                                 │
│        build-and-test ✅                                     │
│        ├─ Checkout → Node.js → Install → Lint              │
│        ├─ Test → Build                                      │
│        └─ Upload artifact ✅                                │
│                                                             │
│        deploy ✅                                             │
│        ├─ Download artifact                                 │
│        ├─ Upload to server                                  │
│        └─ Restart application                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 设计原则总结

| 原则 | 说明 |
|------|------|
| **测试先行** | 代码必须通过测试才能部署 |
| **构建验证** | 代码必须能正常构建才能部署 |
| **PR 保护** | Pull Request 只测试不部署，防止错误代码进入生产 |
| **自动部署** | 代码合入主分支后自动部署，快速交付 |
| **环境隔离** | 构建在 CI 环境，部署在生产服务器 |
| **一次构建** | 构建一次，使用 artifact 传递，避免重复构建 |

### 核心价值

| 价值 | 说明 |
|------|------|
| ✅ **安全** | 防止未经审查的代码进入生产 |
| ✅ **稳定** | 生产环境不会被频繁变动 |
| ✅ **高效** | 审查通过后自动部署，无需人工干预 |
| ✅ **透明** | 每次部署都有完整的记录和日志 |

## 附录：常见问题排查

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| 应用启动失败 | 端口被占用 | `lsof -i :3000` 查看并关闭占用进程 |
| 数据库连接失败 | 配置错误 | 检查 `.env.production` 中的数据库配置 |
| 404 错误 | Nginx 配置问题 | 检查 `proxy_pass` 是否正确 |
| 502 Bad Gateway | 应用未启动 | `pm2 status` 检查应用状态 |
| 证书过期 | Certbot 未续期 | `sudo certbot renew` |
